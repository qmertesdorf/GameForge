# M1.5 — Raster asset method (local SD) design

**Date:** 2026-05-31
**Status:** approved (brainstorming) — pending implementation plan
**Predecessor:** M1 (`asset` skill, SVG re-skin) — see `docs/superpowers/specs/2026-05-30-m1-asset-skill-design.md`
**Research basis:** memory `local-genai-stack-research` (verified May 2026, RTX 3070 8GB)

## 1. Goal

**North star (the end state this milestone serves):** one `asset` skill that can re-skin **a multitude of games across genres and across distinctly different art styles, each game internally cohesive, with output that is genuinely app-store-ready** (correct on every mobile density bucket). M1.5 does not deliver all of that — but it must put the system credibly on that path and make no choice that blocks it.

**M1.5's concrete proof:** the pipeline can produce **representational raster art** (a character/creature/textured look) that hand-authored SVG cannot do well, in **≥2 visibly distinct art styles**, as a sharp reusable **system** whose every gap is **legibly attributable** to specific skill (or tool) prose, exactly as the POC and M1 required.

The deliverable is a **better re-skin system**, not the prettier game. "This sprite looks bad" must always trace to a fixable cause: a weak prompt scaffold, a wrong model/style/param, a mis-placed sprite, a left-in primitive, or an infra failure in the generation tool — never an unattributable blob.

This cycle is **raster art only**. Audio (SFX/music via a local model) is decomposed out to a later **M1.6** cycle; full store-readiness (icon, splash, screenshots, packaging, texture budget) is decomposed to a named **M2** milestone (§12). The two share almost no tooling with raster generation and a combined spec would sprawl.

## 2. Pipeline placement

The pipeline shape is unchanged. Raster is a **second method** of the existing post-`playable` bolt-on:

```
concept → builder → validator → [playable] → asset(method: svg | raster) → validator(re-run) → [styled]
```

The `asset` skill **branches on `concept.art_direction`**:

- geometric / neon / flat / hyper-casual / UI → **`svg`** (M1, unchanged).
- representational / character / illustrated / textured → **`raster`** (M1.5).

This branch criterion is the *same* "SVG aesthetic boundary" the M1 skill already documents. Today the skill says, when it hits a representational concept, "SVG would be mediocre here — M1.5 would serve it better." M1.5 gives the skill somewhere to **go** instead of merely flagging.

## 3. Architecture

**One skill, two methods, shared machinery.** The existing `asset` skill keeps ownership of all the rewiring craft, which is *method-agnostic*: once a texture is on disk, the swap is identical whether it came from authored SVG or a generated PNG. Shared, unchanged:

- `_draw()` → `Sprite2D` (world) / `TextureRect` (HUD), or the preferred `draw_texture*`-in-place for immediate-mode games.
- Footprint placement & scaling to the primitive's original transform.
- Removing/guarding the matching primitive (no double-draw).
- Screen-shake hoisting, per-actor pooling, `modulate` tinting (the run-007 lessons).
- Manifest recording and hand-off to `validator` (the skill never sets `styled` itself).

**Raster-specific additions (the only new surface):**

1. A **generation step** — turn a recipe (prompt + params) into a committed RGBA PNG via the new `tools/comfy.mjs`.
2. **Provenance recording** — store the recipe in the manifest.

## 4. The generation seam

### 4.1 Local stack (prerequisite infra)

- **ComfyUI**, run headless as a local server on the RTX 3070, exposing its HTTP/WebSocket API (`/prompt`, `/history/<id>`, `/view`, `/ws`).
- **SDXL** checkpoint (base 1.0 or a sprite-friendly SDXL finetune). Fits 8GB in fp16 with ComfyUI smart offload. **GGUF is not required** for SDXL — GGUF was a FLUX/SD3.5-on-8GB concern, not relevant here.
- **ComfyUI-layerdiffuse** node — produces true **RGBA** sprites (native alpha at generation time). This is why the stack is locked to SDXL: LayerDiffuse does not support FLUX/SD3.5.
- Optional: a pixel-art / sprite **LoRA** if `art_direction` calls for it.

ComfyUI is treated like the Godot binary: **assumed installed and running**, started by the owner (or a documented `!` launch command), not managed by the skill. The skill/tool only talk to its API.

### 4.2 `tools/comfy.mjs` (new)

A small Node tool, mirroring `tools/manifest.mjs`, that is the **testable seam** between a recipe and a PNG on disk:

```
node tools/comfy.mjs --check                          # ping ComfyUI; report reachable + models present
node tools/comfy.mjs gen <id> <asset-name> '<recipe-json>'
```

`gen` behavior:

1. Load a **workflow-JSON template** (an exported ComfyUI graph with placeholder fields) from a `templates/` directory.
2. Inject recipe fields: `prompt`, `negative`, `seed`, `steps`, `cfg`, `sampler`, `master_resolution` (the high-res master, §5.4), `layerdiffuse` (on/off), `checkpoint`, `lora`.
3. `POST /prompt`, poll `/history/<prompt_id>`, download the output image via `/view`.
4. Write `games/<id>/art/<asset-name>.png`.
5. On unreachable ComfyUI or a graph error, **fail loudly with a clear message** so the failure is attributable to *infra*, not the skill's art judgment.

**Why a tool, not prose:** generation is deterministic plumbing and belongs in testable code (HTTP layer mockable in vitest), leaving `SKILL.md` to own *art judgment* — the visual system, the prompt scaffold, what to re-skin vs. leave. Same division of labor as `manifest.mjs` vs. the skills.

## 5. The `raster` method inside `asset`

### 5.1 Step 0 — visual system first, with style as a first-class choice

Before generating anything, derive one coherent visual system from `concept.art_direction`: palette with named roles, form language, shading model, scale/padding. For raster, Step 0 **additionally** does two things:

**(a) Select the art style explicitly (per game).** Art style is a **first-class, per-game parameter**, not a fixed assumption — this is what lets the *one* skill make *different-looking* games. From `art_direction`, the skill chooses a **style profile**: which SDXL **checkpoint** (e.g. a pixel-art finetune vs. a painterly vs. a flat-cartoon checkpoint), which **LoRA(s)** if any, and the **style fragment** of the prompt. The chosen profile is recorded in `asset_pass.visual_system.style` so a different game can deliberately pick a different one. The skill must be able to justify "this `art_direction` → this style profile" the same way it justifies the SVG visual system today.

**(b) Fix the shared prompt scaffold.** A base style string (from the chosen profile) + per-actor subject, plus a fixed sampler/steps/cfg, so every sprite in *this* game is generated from one scaffold and reads as a single designed set.

"Each sprite is fine but they don't cohere" is the same *primary failure* M1 guards against — prevented here by the style profile + shared scaffold + fixed params, not by independent per-actor prompts. Incoherence within a game is a finding about the **scaffold/profile**, not the individual PNGs. (Conversely, two *different games* should look deliberately different because they selected different profiles — that difference is the goal, not a failure.)

### 5.2 Per-entity flow

1. **Recipe** — compose prompt from `scaffold + this actor's subject`, plus negative prompt, seed, and params.
2. **Generate** — `node tools/comfy.mjs gen <id> <name> '<recipe>'` → RGBA PNG in `art/`.
3. **Import** — `godot --headless --path games/<id>/ --import` (creates `.png.import` sidecars + cached textures *before* re-validation).
4. **Rewire** — identical to M1: `draw_texture*`-in-place (preferred for the builder's default single-`_draw()` games) or pooled `Sprite2D`/`TextureRect` nodes; place at the primitive's original footprint; **remove/guard** the matching primitive; **game logic untouched**; verify no double-draw.

### 5.3 Mixed-method honesty (inverted boundary)

M1's `asset` flags when representational art needs raster. M1.5 inverts it: if an actor is genuinely better as crisp vector or primitive (a UI bar, a geometric pickup, a HUD frame), the skill is **allowed to leave it `svg` or `primitive`** and must say so. A run can be **mixed-method**. Record which entities are `raster` vs `svg` vs `primitive`, so a partial raster pass is a legible choice, not a silent gap.

### 5.4 Resolution & mobile density (recovering what SVG gave for free)

SVG gave every Android density bucket (mdpi → xxxhdpi) from one file — an explicit app-store strength M1 leaned on. Raster forfeits that, so M1.5 must replace it deliberately or the output is **not** app-store-ready:

- **Generate high-res masters.** Each sprite is generated at a master resolution sized for the **largest** density bucket it will occupy on screen (a generous power-of-two master, e.g. 512² or 1024² for a hero actor, smaller for minor props), recorded in the recipe. We rely on **downscale-from-master**, never upscale, so the asset stays crisp on xxxhdpi and Godot scales down for lower densities.
- **Godot mobile import settings.** The `.png.import` sidecar is configured for mobile: mipmaps **on** (clean minification), an appropriate filter, and VRAM/lossless compression suited to 2D mobile. The skill sets these explicitly rather than accepting Godot's editor defaults, and records the choice.
- **Footprint mapping unchanged.** As with SVG, the sprite is scaled to the primitive's on-screen footprint at runtime; the master being larger than the footprint is expected and correct (headroom for high-DPI), not waste to "fix" by shrinking the master.

This keeps M1.5 output genuinely mobile-grade. Texture **atlasing** and APK **size budgeting** across a whole title are a packaging concern and belong to **M2** (§12), not here — but generating proper masters now is what makes that later step possible.

### 5.5 Content & IP safety

SDXL can emit outputs that resemble trademarked/copyrighted characters, which is an app-store rejection and legal risk. The skill must: include a **negative prompt** discouraging known IP / logos / celebrity likeness, keep prompts to **generic descriptors** (no named franchises), and treat the **human A/B** as the safety review (the owner confirms nothing looks like protected IP before `styled`). Licensing of the models/outputs themselves is settled in the research note (SDXL Community License; outputs owned, free commercial under the revenue threshold) — re-verify thresholds at ship time.

## 6. Schema / manifest changes (additive, non-breaking)

All additions are optional; existing manifests remain valid. `additionalProperties:false` requires explicit declaration in `schema/manifest.schema.json`.

- `assets[].origin` — gains the value `"raster"` (field is already free-text string; no schema change needed, documented value).
- `asset_pass.method` — gains the value `"raster"` (free-text string; documented value).
- `asset_pass.visual_system.prompt_scaffold` — **new** optional string: the shared style skeleton.
- `asset_pass.visual_system.style` — **new** optional object: the selected per-game **style profile** — `{ checkpoint, loras[], style_prompt }` (§5.1a). This is the field that makes "different art styles" legible and reusable.
- `asset_pass.recipes[]` — **new** optional array, one entry per generated PNG:
  `{ name, checkpoint, prompt, negative, seed, sampler, steps, cfg, master_resolution, layerdiffuse, lora, import_settings }`.
  `master_resolution` is the high-res master size (§5.4); `import_settings` records the mobile `.png.import` choices (mipmaps/filter/compression). This is the committed provenance.

`prompt_scaffold`, `style`, and `recipes[]` are the structural schema additions; the `origin`/`method` changes are just new values of existing string fields.

## 7. Determinism & git

- The committed **PNG is canonical** — the game runs from a fresh clone exactly as M1's SVGs do. PNGs and their `.png.import` sidecars are committed.
- `recipes[]` is **provenance, not a bit-exact regeneration guarantee.** GPU matmul + RNG are non-deterministic (confirmed in research); the same recipe regenerates a *close* image, never the same pixels. The skill states this plainly so "I have the seed" is never mistaken for "I can reproduce the art."

## 8. Feasibility gate (first implementation phase)

Because no throwaway spike was run, feasibility is folded into the build as an explicit **early gate, before any proof run**:

- Stand up ComfyUI + SDXL + layerdiffuse on the 3070.
- `tools/comfy.mjs --check` succeeds, and one test `gen` produces **one usable RGBA sprite** at a high-res master size (§5.4).
- **Confirm style-switching works** — generate a second sprite under a *different* style profile (a different checkpoint and/or LoRA), since multi-style (§5.1a, §9) depends on swapping profiles. This surfaces checkpoint disk/load-time cost and any per-profile VRAM ceiling early.
- If VRAM, edge quality, throughput, or profile-switching disappoint, we learn it here — attributable to *infra/stack*, before investing in the full proof. This retires the research's "unverified on real hardware" risk.

## 9. Proof runs (≥2, style-distinct) — run-008 & run-009

To demonstrate "**different** art styles," M1.5 proves on **two games whose `art_direction` selects visibly distinct style profiles** (§5.1a) — e.g. a painterly/illustrated creature game and a flat-cartoon character game (or pixel-art) — not one. Each exercises the whole path:

```
prompt (representational art_direction, style A)
  → concept → builder → validator → owner playtest → [playable]
  → asset(raster) → validator(re-run) → owner A/B → [styled]
  → docs/superpowers/poc-run-008.md
prompt (representational art_direction, style B)  → … → poc-run-009.md
```

The two runs share the **same skill and tool, unchanged** — only the per-game style profile differs. That is the core evidence: one system, two deliberately different looks, each internally cohesive. Each report ties gaps to specific prose (skill or `comfy.mjs`), exactly like runs 001–007, and run-009 additionally records whether findings from run-008 carried forward (the compounding-findings pattern). The two concepts (a character-forward genre per style) are proposed in the implementation plan.

A second style-distinct proof is the **minimum** to claim style generality; the plan may sequence run-009 after run-008's findings are folded in, so the second run also tests that the edited prose holds.

## 10. Testing

- `tools/comfy.mjs` — vitest with the **HTTP layer mocked**: recipe→workflow injection, polling `/history`, `/view` download, and error paths (unreachable, graph error). No GPU in CI; consistent with the all-JS suite.
- Schema additions — covered by the existing manifest validation tests (new fields accepted; existing manifests still valid).
- The art itself — gated by the **human A/B playtest**, like every run. Not automatable.

## 11. M1.5 scope (what this cycle delivers)

In scope, full stop:

- The `raster` method inside `asset`: representational **RGBA sprites via SDXL + LayerDiffuse**.
- **Art style as a first-class per-game parameter** (style profiles), proven across **≥2 visibly distinct styles**.
- **`tools/comfy.mjs`** — the tested recipe→PNG seam against a local ComfyUI server.
- **Mobile resolution/density** done right (high-res masters + explicit mobile import), so output is app-store-grade per asset.
- Additive schema (`style`, `prompt_scaffold`, `recipes[]`); IP-safety guardrail.
- Two style-distinct proof runs (run-008, run-009) + reports.

Explicitly **not** in M1.5 (each lands in a named milestone below, so deferral is tracked, not silent):

- **Audio** (SFX/music) → **M1.6**.
- **Animation / sprite sheets, backgrounds & tiles, more asset types** → staged raster follow-ons (see §12).
- **Store packaging** (app icon + sizes, splash, screenshots, texture atlasing, APK size budget, export presets) → **M2**.
- **FLUX/SD3.5/GGUF, non-LayerDiffuse transparency** → not needed for this stack; revisit only if a style profile demands it.

## 12. Roadmap toward the north star (so the goal is tracked, not assumed)

The §1 north star — *many genres × many art styles, each cohesive, fully app-store-ready* — is reached across these named milestones. M1.5 makes the choices that keep them all reachable (per-game style profiles, proper masters) without building them yet.

- **M1.5 (this spec):** representational RGBA sprites, multi-style, mobile-grade per asset.
- **M1.6 — audio:** SFX/music via a local model; same legible-attribution discipline.
- **M1.7 — animation & backgrounds (staged):** multi-frame sprite sheets / animation and representational + tiling backgrounds, as builder output grows beyond procedural motion. Broadens *asset-type* coverage across genres.
- **M2 — store-readiness:** app icon (+ all required sizes), splash, store screenshots, texture **atlasing**, **APK size budget**, Godot Android **export presets** — the title-level packaging that turns a folder of mobile-grade assets into a shippable build.

App-store readiness is therefore **not deferred away** — it is the M2 destination this milestone is deliberately paving toward, and M1.5's per-asset density work is its first concrete down payment.
